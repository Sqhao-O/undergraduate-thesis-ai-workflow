## 2. 阶段一：需求整理（Kimi Agent）

### 2.1 目标

明确毕业论文的：
- 研究方向和题目
- 技术路线和方案
- 功能模块划分
- 预期成果和创新点

### 2.2 使用工具

**Kimi Agent**（https://kimi.moonshot.cn/）

> 为什么用 Kimi？因为它的中文理解能力强、上下文窗口大，适合长文本的需求梳理。

### 2.3 操作步骤

```
1. 打开 Kimi 网站（需要登录）
2. 如果有 Agent 模式，切换到 Agent 模式
3. 在对话框中输入你的初步想法，例如：

   "我要做一个基于深度学习的图像识别系统作为本科毕业论文，
    帮我整理一份完整的需求文档框架，包括研究背景、技术选型、
    功能模块设计、预期成果等。"

4. 根据 Kimi 的输出进行多轮对话，逐步细化需求
5. 将最终确定的内容保存为 Markdown 文件（requirements.md）
```

### 2.4 输出产物

```markdown
# requirements.md 示例结构

## 论文题目
基于 XXX 的 YYY 系统研究与实现

## 研究背景
（AI 帮你整理的背景资料）

## 技术路线
- 方案 A：...
- 方案 B：... （推荐）
- 选择理由：...

## 功能模块
1. 数据预处理模块
2. 核心算法模块
3. 用户界面模块
4. ...

## 创新点（必须自己想！！！）
1. ...
2. ...

## 参考文献
（AI 帮你搜集的）
```

### 2.5 注意事项

- 让 AI 帮你整理和归纳
- 用 AI 扩展阅读面
- 不要让 AI 替你决定创新点——这必须是原创！

### 2.6 使用 brainstorming Skill 辅助构思

除了 Kimi Agent，还可以用 Claude Code 的 brainstorming Skill 帮你理清研究方向。

#### 2.6.1 brainstorming 适用场景

| 场景 | 表现 | 适合的工具 |
|------|------|-----------|
| **需求明确** | 已经知道要做什么 | 直接用 Kimi Agent 写需求文档 |
| **需求模糊** | 大概知道方向但不具体 | brainstorming Skill 帮你头脑风暴 |

#### 2.6.2 如何使用 brainstorming

```
1. 在 Claude Code 中安装 Skill：
   /skills add brainstorming

2. 启动 brainstorming：
   请帮我分析 [你的研究方向] 的研究现状，
   提出 3-5 个可行的研究点

3. 根据输出确定具体课题

4. 再用 Kimi Agent 或直接让 Claude Code 写需求文档
```

#### 2.6.3 示例：需求不明确时

假设你只知道想做"机器学习相关"，但不确定具体做什么：

```
> 请帮我头脑风暴：我对机器学习和数据分析感兴趣，
  但不确定具体做什么方向。能推荐几个适合本科毕业论文的
  研究方向吗？最好是比较新颖且有实际应用价值的。
```

brainstorming 会帮你：
- 分析该方向的研究现状
- 提出多个可行的研究点
- 评估每个方向的难度和可行性
- 最终帮你确定具体课题

#### 2.6.4 输出产物

- 明确的研究方向和题目
- 需求文档 `requirements.md`
- 后续可以直接进入阶段二（代码开发）

## 3. 阶段二：代码开发与仿真（Claude Code）

### 3.1 目标

根据需求文档，完成：
- 项目工程搭建
- 核心代码编写
- 代码调试与测试
- 仿真/实验运行
- 结果数据收集

### 3.2 使用工具

**Claude Code**（详见 [Claude Code 配置教程](./01-claude-code-setup.md)）

### 3.3 操作步骤

```
1. 打开终端（CMD）
2. 进入项目文件夹：
   cd d:\你的项目路径
3. 启动 Claude Code：
   claude
4. 让它读取需求文档：
   > 请读取 requirements.md 文件，理解项目的整体需求
5. 分步让 Claude Code 实现：
   > 请先搭建项目的基础目录结构
   > 请实现数据预处理模块
   > 请编写核心算法代码
   > 请帮我运行测试，看看有没有错误
6. 运行仿真/实验：
   > 请运行 main.py 并分析输出结果
7. 收集实验数据和结果截图
```

### 3.4 与 Claude Code 对话的最佳实践

#### 3.4.1 好的做法

```
> 请你读取当前文件夹下的《需求文档.md》，根据文档里的需求，完整编写可直接运行的代码，创建对应的工程文件，代码要添加详细注释，符合本科毕设的规范要求。

> 我在运行时遇到了 TypeError: ... 错误，
  请帮我定位原因并修复。

> 请在 tests/ 目录下创建单元测试文件，
  测试 core.py 中的所有公开函数。
```

#### 3.4.2 不好的做法

```
> 帮我写个毕设
> 你说该怎么做？
> 我的程序出错了（但不给具体信息）
```

### 3.5 输出产物

```
项目文件夹/
├── src/                 ← 源代码
│   ├── main.py
│   ├── simulator.py
│   ├── utils.py
│   └── ...
├── data/                ← 数据文件
│   ├── input/
│   └── output/          ← 仿真结果数据
├── tests/               ← 测试代码
├── results/             ← 实验结果和图表
│   ├── fig1_accuracy.png
│   ├── fig2_loss.png
│   └── data_results.csv
├── requirements.txt     ← Python 依赖
└── README.md            ← 项目说明
```

## 5. 阶段四：格式转换（Pandoc）

### 5.1 目标

将 Markdown 格式的论文转换为学校要求的 DOCX（Word）格式。

### 5.2 使用工具

**Pandoc**（详见 [Pandoc 使用教程](./03-pandoc-usage.md)）

### 5.3 操作步骤

```
1. 确保已安装 Pandoc：
   pandoc --version

2. 准备学校的 Word 模板文件（template.docx）

3. 执行转换命令：
   pandoc thesis.md -o thesis.docx \
     --reference-doc=template.docx \
     --toc \
     --toc-title="目  录"

4. 双击打开 thesis.docx，检查转换效果
```

### 5.4 转换后检查清单

- [ ] 所有章节标题格式正确
- [ ] 图片正常显示
- [ ] 表格正常显示
- [ ] 公式正常显示
- [ ] 目录自动生成
- [ ] 字体基本正确（后续工具会微调）
- [ ] 页码位置正确

## 6. 阶段五：内容完善（人工修改）

> ⚠️ **重要提醒**：这一步没有 AI 参与，完全靠你自己！AI 生成的只是"草料"，你需要把它变成"菜肴"。你的独立思考和分析是论文的灵魂，学术诚信要求所有核心内容必须是原创。

> 💡 **修改原则**：用你自己的话重写AI生成的内容、补充真实的分析和见解、确保所有数据真实、保持语言风格统一。

### 6.1 使用 Claude Code + Skills 辅助润色

除了人工修改，还可以用 Claude Code 配合以下 Skills 进行辅助润色。

#### 6.1.1 academic-paper-reviewer — 学术论文评审

这是专门用于检查学术论文的 Skill，可以帮你：
- 检查论文逻辑结构和连贯性
- 指出语言表达问题
- 提出修改建议

```
1. 安装 Skill：
   /skills add academic-paper-reviewer

2. 使用 Skill 检查论文：
   请检查论文的逻辑结构、语言表达、格式规范

3. 根据建议进行修改
```

#### 6.1.2 pdf Skill — 处理 PDF 文献

在润色阶段，可能需要回顾和引用 PDF 格式的文献：

```
1. 安装 Skill：
   /skills add pdf

2. 读取 PDF 文献：
   请读取并总结这篇 PDF 的主要内容

3. 提取关键信息用于论文
```

#### 6.1.3 docx Skill — 处理 Word 文档

用于检查和修改论文格式：

```
1. 安装 Skill：
   /skills add docx

2. 检查论文格式：
   请帮我检查论文格式，并生成修改建议

3. 应用修改
```

#### 6.1.4 注意事项

- AI 润色只是辅助，核心内容和观点必须自己完成
- 润色后务必通读全文，确保意思没有偏离
- 参考文献和引用务必核实真实性

### 6.2 学术规范提醒

**Q: AI生成的内容可以直接提交吗？**

**A:** **不可以！** AI只能作为辅助工具，你需要：
1. 用自己的话重写AI生成的内容
2. 添加自己的分析和见解
3. 确保实验数据真实可靠
4. 检查所有引用是否真实存在

**Q: 如何正确引用AI工具？**

**A:** 不同学校要求不同，一般建议：
1. 在致谢中说明使用了AI工具辅助
2. 在方法部分说明AI的具体用途
3. 咨询导师或查阅学校的学术规范文件

## 7. 阶段六：初步排版（Lark-Formatter）

### 7.1 目标

对 DOCX 论文进行一键自动化排版，处理基本的格式问题。

### 7.2 使用工具

**Lark-Formatter**  
**GitHub：** https://github.com/Alouetter/Lark-Formatter  
**当前版本：** 0.2.1 Hotfix（基于 0.20 LTS）

### 7.3 操作步骤

```
1. 下载 Lark-Formatter
   GitHub Releases 页面下载最新版本

2. 解压到任意目录（如 D:\Tools\Lark-Formatter）

3. 启动程序：
   - Windows: 双击 start_app.bat
   - 或者终端中运行 start_app.bat

4. 加载论文文件：
   - 点击"打开文件"或拖拽 thesis.docx 到窗口

5. 配置排版选项（根据学校要求）：
   - 正文字体、字号
   - 标题字体、字号
   - 行距设置
   - 页边距设置
   - 标题编号方式

6. 点击"一键排版"

7. 检查排版结果：
   - 标题编号是否正确（1, 1.1, 1.1.1...）
   - 字体是否统一
   - 行距是否合适
   - 页边距是否正确

8. 导出/保存排好版的文件
```

### 7.4 Lark-Formatter 能做什么？

| 功能 | 说明 |
|------|------|
| 多级标题编号 | 自动处理一级/二级/三级标题编号 |
| 字体设置 | 统一正文、标题字体和字号 |
| 行距调整 | 设置固定行距或倍数行距 |
| 页边距 | 统一设置上下左右页边距 |
| 段落缩进 | 首行缩进两个字符 |
| 页眉页脚 | 添加页码等信息 |

### 7.5 排版后检查

- [ ] 一级标题："第一章"、"第二章"... 编号正确
- [ ] 二级标题："1.1"、"1.2"... 编号正确
- [ ] 三级标题："1.1.1"、"1.1.2"... 编号正确
- [ ] 正文字体字号符合学校要求
- [ ] 行距符合要求（通常 1.25 或 1.5 倍）
- [ ] 页边距符合要求

### 8.1 目标

使用 WorkBuddy 的 Paper Check Skill 进行精细化的格式检测和修复。

### 8.2 使用工具

**WorkBuddy（腾讯小龙虾）下载地址：** https://www.codebuddy.cn/work/  
**Paper Check Skill 下载地址：** https://clawhub.ai/sashavegal/paper-check  

详细操作指南：[WorkBuddy 排版教程](./04-workbuddy-guide.md)

### 8.3 核心流程概要

```
1. 安装并启动 WorkBuddy（Claw 模式）
2. 下载 Paper Check Skill
3. 创建 skill 文件夹，放入论文和格式规范
4. 初始化 Skill，读取排版要求
5. 自动检测格式问题
6. 一键修复或手动确认修复
7. 生成对比报告
8. 导出最终版本
```

### 8.4 WorkBuddy vs Lark-Formatter 的区别

| 维度 | Lark-Formatter | WorkBuddy |
|------|----------------|-----------|
| **类型** | 桌面应用程序 | AI Agent 平台 |
| **原理** | 规则引擎（固定规则） | AI 理解（智能判断） |
| **优势** | 快速、批量处理 | 更灵活、能理解复杂规范 |
| **擅长** | 标题编号、字体行距 | 复杂格式、细节检查 |
| **推荐用途** | 初步排版 | 精细调整 |

### 8.5 最佳实践

建议 **两者结合使用**：

```
Markdown 论文初稿
    ↓
  Pandoc 转换为 DOCX 格式
    ↓
  Lark-Formatter（初步排版：标题编号、字体、行距）
    ↓
  人工内容完善
    ↓
  WorkBuddy（精细检查：细节格式、特殊要求）
    ↓
  最终版论文 ✓
```

## 9. 阶段七（续）：PPT 制作（pptx Skill）

### 9.1 目标

使用 Claude Code 的 pptx Skill 制作毕业答辩 PPT。

### 9.2 什么时候做 PPT？

**PPT 是在论文全部完成后才开始**，不是在写论文的同时做。

建议时间线：
- 论文初稿完成 → 开始制作 PPT
- PPT 内容基于论文，但更精简
- 答辩前一周完成 PPT，之后只做小幅调整

### 9.3 使用工具

**pptx Skill** — Claude Code 的 PPT 制作技能

```
/skills add pptx
```

详细 PPT 制作指南见下文。

### 9.4 pptx Skill 核心功能

| 功能 | 说明 |
|------|------|
| 创建 PPTX 文件 | 从头创建新的演示文稿 |
| 添加幻灯片 | 插入空白或带布局的幻灯片 |
| 设置布局 | 指定标题、内容、图片等占位符 |
| 插入内容 | 文字、图片、图表、形状 |
| 批量修改样式 | 统一字体、颜色、对齐方式 |

### 9.5 快速开始

```
/skills add pptx
请帮我制作一个毕业答辩 PPT，包含以下内容：
1. 封面（题目 + 姓名 + 指导教师）
2. 研究背景与意义
3. 研究内容与方法
4. 实验结果与分析
5. 结论与展望
```

### 9.6 答辩 PPT 制作规范

#### 9.6.1 配色方案

| 推荐配色 | 说明 |
|----------|------|
| 深蓝 + 白色 + 灰色 | 专业、稳重，适合学术答辩 |
| 深蓝 | RGB: 0, 82, 147 或类似 |

| 避免使用 | 原因 |
|----------|------|
| 过于鲜艳的颜色 | 显得不专业 |
| 花哨渐变 | 分散注意力 |

#### 9.6.2 字体规范

| 元素 | 字体 | 样式 | 字号 |
|------|------|------|------|
| 标题 | 微软雅黑 / 思源黑体 | 加粗 | 32-44pt |
| 正文 | 微软雅黑 | 常规 | 24-28pt |
| 辅助文字 | 微软雅黑 | 常规 | 18-20pt |

#### 9.6.3 布局原则

- 每页不超过 **6 行**，每行不超过 **6 个词组**
- 图文结合，图表比文字更有说服力
- 适当留白，不要太拥挤
- 重点内容突出显示

#### 9.6.4 推荐结构（7-10 页）

| 页码 | 内容 | 建议时长 |
|------|------|----------|
| 1 | 封面（题目 + 姓名 + 指导教师 + 学校） | 30 秒 |
| 2 | 目录 | 15 秒 |
| 3 | 研究背景与意义 | 2 分钟 |
| 4 | 研究内容与方法 | 3 分钟 |
| 5 | 实验结果与分析 | 3-4 分钟 |
| 6 | 结论与展望 | 1-2 分钟 |
| 7 | 致谢 | 15 秒 |

#### 9.6.5 避免的问题

| 问题 | 后果 |
|------|------|
| 文字堆砌，念 PPT | 评委失去兴趣 |
| 动画过多过花哨 | 显得不专业 |
| 图表不清晰、不标注来源 | 影响说服力 |
| 超过答辩时间限制 | 影响整体评分 |

### 9.7 PPT 制作流程

```
1. 准备好论文的核心内容（结果、图表、结论）
2. 调用 /skills add pptx 启用 PPT 技能
3. 让 Claude Code 根据论文内容生成 PPT 大纲
4. 逐页确认内容是否准确
5. 导出 PPTX 文件
6. 用 PowerPoint 打开检查效果
7. 根据需要手动调整细节
```

## 10. 阶段八：最终校对与提交

### 10.1 最后的人工检查

这是最后一道防线！请逐项检查：

#### 10.1.1 内容检查

- [ ] 通读全文至少 **2 遍**
- [ ] 检查所有**错别字**
- [ ] 检查**语句通顺性**
- [ ] 检查**逻辑连贯性**
- [ ] 确认**创新点**表述清晰
- [ ] 确认**结论**与研究内容对应

#### 10.1.2 格式检查

- [ ] **封面信息**正确（姓名、学号、指导教师等）
- [ ] **摘要**中英文对照
- [ ] **关键词**数量和格式正确
- [ ] **目录**页码与正文一致
- [ ] **图表编号**连续且引用正确
- [ ] **参考文献**格式统一（GB/T 7714）
- [ ] **页眉页脚**正确
- [ ] **页码**从正文开始

#### 10.1.3 文件准备

- [ ] 导出 **PDF** 版本（学校一般要求 PDF 提交）
- [ ] 保留 **DOCX** 版本（备用）
- [ ] 准备**源代码**压缩包（如果需要）
- [ ] 准备**答辩PPT**（另外制作）

### 10.2 提交前自检表

```
□ 论文查重率在学校规定范围内（一般 < 30%）
□ 参考文献数量达到要求（一般 >= 15篇）
□ 所有实验数据可复现
□ 导师已经审阅过并给出修改意见
□ 所有导师意见已修改完成
□ 文件命名符合学校要求
□ PDF 可以正常打开和打印
```
